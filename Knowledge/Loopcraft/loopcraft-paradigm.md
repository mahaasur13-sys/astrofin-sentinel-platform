# Loopcraft — проектирование замкнутых автономных циклов

> *«Проектирование цикла означает: задать повторяющееся действие, метрику прогресса, механизм автоматической оценки и условие остановки. Человек становится архитектором правил, а не писателем кода.»*
>
> — из обсуждения loopcraft, лето 2026

---

## 1. Суть loopcraft и сдвиг парадигмы

Традиционная работа с нейросетью — **линейная**: человек пишет запрос, модель возвращает ответ, человек применяет его (или отбрасывает). **Loopcraft** предлагает другую модель: человек задаёт **замкнутый цикл**, внутри которого агент сам генерирует действие, проверяет результат автоматической верификацией и повторяет попытку до достижения критерия остановки.

Аналогия из обсуждения: **клиент в ресторане** — заказал суп, попробовал, вернул повару, тот исправил и подал заново, клиент снова пробует. **Повар** здесь — агент, **критика клиента** — функция верификации, **счёт попыток** — бюджет цикла.

Смещение ролей:

| Роль | До loopcraft | В loopcraft |
|---|---|---|
| Человек | Пишет код, отлаживает | Задаёт правила, метрики, шлюзы |
| Агент | Одноразовый исполнитель | Замкнутый цикл «попытка → проверка → коррекция» |
| Критерий успеха | «Работает на моей машине» | Формальная метрика + автоматический тест |
| Время инженера | Написание кода | Проектирование верификации |

## 2. Пример: Autoresearch (Андрей Карпати)

Конкретный эталон, на который ссылается обсуждение:

1. Агент получает `train.py` как редактируемый артефакт.
2. Вносит правку (гиперпараметр, токенизация, шедулер).
3. Запускает обучение.
4. Замеряет **валидационную метрику**.
5. **Если улучшение** — коммит и сохранение.
6. **Если ухудшение** — `git checkout` к предыдущему SHA, новая попытка.
7. Цикл повторяется, пока не исчерпан бюджет итераций или не достигнута target-метрика.

**Ключевая деталь**: человек не пишет код улучшения — он пишет **обвязку** (раннер, метрику, gate, откат). Это и есть loopcraft-проектирование.

## 3. Иерархия уровней автономности

Из обсуждения выделена **5-уровневая шкала**:

| Уровень | Название | Что делает | Пример |
|---|---|---|---|
| **1** | Базовый LLM | Генерирует токены, не видит результат | `chatgpt.com` обычный чат |
| **2** | Tool-use без верификации | Вызывает инструменты, но не проверяет эффект | Агент вызывает `pip install`, не проверяя импорт |
| **3** | Цикл с обратной связью | Попытка → замер метрики → коррекция | **Autoresearch**, RAG-агенты с self-eval |
| **4** | Метацикл (цифровой дарвинизм) | Система сама порождает новых агентов при провалах | AutoScientists (Harvard), GEPA-style оптимизаторы |
| **5** | Командный цикл (рой) | Специализированные агенты взаимодействуют в многослойном цикле | Anthropic Multi-Agent Research (Opus 4 +90,2% над single-agent) |

> Loopcraft начинается с **уровня 3** и масштабируется вверх.

## 4. Экономика и ограничения

Главное эмпирическое наблюдение из обсуждения: **агенты в циклах потребляют в 4–15 раз больше токенов**, чем одноразовый вызов. Причины:

- **Растущий контекст** — каждая итерация добавляет историю попыток, логи, состояние.
- **Самокритика** — отдельный проход модели на оценку своего же ответа.
- **Retry-накладные расходы** — откат + пересборка + повторный замер.

### Ловушка Ральфа Вигома

> *«Без шлюза верификации цикл рапортует об успехе, даже если результат нерабочий.»*

Цикл без gate-а — это «вечный оптимист»: агент пишет «всё работает», gate не проверяет, цикл радостно коммитит сломанный код. **Gate — не опция, а необходимость.**

## 5. Четыре условия экономической оправданности цикла

Цикл имеет смысл запускать **только** при выполнении **всех четырёх** условий:

1. **Повторяемость задачи** — задача возникает регулярно или допускает много итераций улучшения.
2. **Встроенная автоматическая верификация** — существует быстрый и дешёвый тест, который однозначно отличает хороший результат от плохого.
3. **Бюджет на токены с запасом** — есть ресурс на 4–15× множитель относительно одноразового вызова.
4. **Доступ к инструментам уровня старшего инженера** — агенту доступны shell, git, pytest, профилировщики, IDE-API.

> Нарушение любого из четырёх превращает loopcraft в дорогой рандом.

## 6. Рецепт минимально жизнеспособного безопасного цикла

Из обсуждения — четыре обязательных компонента:

```
┌──────────────────────────────────────────────────────┐
│  1. ТРИГГЕР                                          │
│     Явное событие запуска (cron, webhook, manual)    │
├──────────────────────────────────────────────────────┤
│  2. ИЗОЛИРОВАННЫЙ НАВЫК                              │
│     Один агент с ограниченным контекстом             │
│     (не «универсальный помощник», а «оптимизатор X») │
├──────────────────────────────────────────────────────┤
│  3. STATE-ФАЙЛ                                       │
│     {accepted: [...], rejected: [...], best: {...}}  │
│     Сохраняется между запусками, не теряется         │
├──────────────────────────────────────────────────────┤
│  4. GATE (шлюз верификации)                          │
│     Автоматический тест, не пропускающий             │
│     нерабочий результат. Без gate — ловушка Вигома.  │
└──────────────────────────────────────────────────────┘
```

## 7. Метрика эффективности: cost per accepted change (CPA)

Обсуждение вводит **главную метрику эффективности цикла**:

> **CPA = (общая стоимость всех попыток) / (количество принятых изменений)**

Это **не** «сколько токенов сожгли» — это **сколько стоил каждый кусок кода, реально попавший в продакшен**.

Пример расчёта:

- 50 попыток × 1000 токенов = 50 000 токенов сожжено
- Принято: 4 улучшения
- **CPA = 12 500 токенов на одно принятое изменение**

Если CPA растёт от итерации к итерации — **цикл исчерпал пространство улучшений**, пора останавливать.

## 8. Смена роли инженера и парадокс тестов

Loopcraft **смещает труд инженера** от кода к верификации:

> *«Смена роли инженера: от написания кода — к проектированию систем верификации и границ.»*

**Парадокс**: написание идеального теста для ИИ порой сложнее, чем написание кода вручную. Это означает, что **качество цикла = качество gate-а**. Слабый gate → цикл бесцельно тратит бюджет. Сильный gate → цикл становится формой автоматизированного code review.

Из обсуждения следует практический вывод: **инвестиция в gate-тесты окупается кратно** — каждый улучшенный gate экономит тысячи токенов на отсеве ложных улучшений.

---

## 9. Связь с другими концепциями

Loopcraft — это **общая рамка**, частные случаи которой уже существуют:

| Конкретный паттерн | Уровень loopcraft | Отличие |
|---|---|---|
| **Autoresearch** (Карпати) | 3 | Конкретный цикл «правка train.py → val_loss → git» |
| **Agent Harness** (Microsoft, 2026) | 3 | Обвязка tool-loop + durable memory + observability |
| **AutoScientists** (Harvard) | 4 | Самоорганизация + failure documentation |
| **Anthropic Multi-Agent Research** | 5 | Рой специализированных агентов |
| **Ralph Wygum trap** | антипаттерн | Цикл без gate → ложный успех |

В моих рабочих проектах (`astrofin-sentinel-v5`, `atom-federation-os`, `roma-execution-bridge`) уже есть элементы уровня 3: `track_agent_metrics` + A/B testing + checkpointing. Loopcraft — это **формализация и расширение** того, что делалось вручную.

## 10. Финальная мысль для обдумывания

> *Loopcraft превращает инженера из автора кода в архитектора правил. Главный навык — не «написать функцию», а «спроектировать шлюз, который отличает правильную функцию от неправильной». Кто освоит это первым в своей предметной области — тот получит асимметричное преимущество: его loop будет работать, пока конкуренты спорят, какую модель вызвать.*

**Вопросы для рефлексии**:
- Какие повторяющиеся задачи в моих текущих проектах можно превратить в loopcraft-цикл?
- Где у меня сейчас gate заменён на «я потом посмотрю» (ловушка Вигома)?
- Чему равен мой CPA по последнему циклу оптимизации? Стал ли он расти?

## Извлечённые концепции, технологии и методы

### Концепции

- **Замкнутый автономный цикл** (closed autonomous loop) — повторяющееся действие с автоматической обратной связью
- **Шлюз верификации** (verification gate) — единственная защита от ловушки Вигома
- **Cost per accepted change (CPA)** — единственная экономически честная метрика цикла
- **Иерархия автономности** — 5 уровней от базовой генерации до роя
- **Цифровой дарвинизм** (уровень 4) — система порождает новых агентов при провалах
- **Смещение роли инженера** — от кода к верификации
- **State-файл** как持久овая память цикла между запусками

### Технологии

- **Autoresearch** (Karpathy) — эталон уровня 3
- **Microsoft Agent Framework v1.0 + Agent Harness** — production-обвязка уровня 3
- **AutoScientists** (Harvard Zitnik Lab, 2026) — самоорганизующаяся система уровня 4
- **Anthropic Multi-Agent Research** — рой уровня 5, +90,2% над single-agent
- **GEPA / prompt evolution** — оптимизаторы промтов уровня 4
- **LangGraph, CrewAI, smolagents, MAF** — четыре основные парадигмы фреймворков 2026
- **Git checkout + SHA-откат** — базовый примитив для level 3 (как в Autoresearch)

### Методы

- **Loopcraft-проектирование** — задать действие + метрику + gate + остановку
- **State-персистентность** — JSON-файл с принятыми/отклонёнными попытками
- **CPA-анализ** — оценка окупаемости цикла
- **Failure documentation as first-class** — журнал неудач как актив (AutoScientists)
- **Изоляция контекста навыка** — один цикл = одна задача, не «универсальный агент»
- **Trigger/Stop-condition** — пара явных условий запуска и остановки
- **Retry с откатом** — git checkout, snapshot restore, транзакционное выполнение
