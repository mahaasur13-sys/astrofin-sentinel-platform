**1. Claude Opus 4.8 + Dynamic Workflows — оркестрация до 1000 параллельных sub-agents из одной сессии**

- **Источник:** Anthropic (релиз Opus 4.8 + Dynamic Workflows research preview) + массовый community traction в X / Reddit / GitHub issues (GitHub anthropics/claude-code)
- **Краткое описание:** Anthropic выкатил Opus 4.8 с принципиально новой фичей — **Dynamic Workflows** (research preview, доступен на Team/Max/Enterprise). Вместо ручного handoff-протокола модель сама пишет orchestration-скрипт, который фан-аутит работу через **до 16 параллельных sub-agents** и **до 1000 инвокаций за один run**. Каждый sub-agent атакует задачу с разных углов, проходит через adversarial-верификацию (отдельные judge-агенты оценивают и отвергают слабые результаты), и только верифицированный финал возвращается в основную сессию. Workflow работает в фоне, главный тред остаётся отзывчивым, поддерживается resume и эффективное управление токенами. Реальный кейс: Jared Sumner пересобрал ~750 000 строк кода за 11 дней, 99.8% тестов прошли. Open-source community уже выпустил порты: `nemori-ai/langchain-dynamic-workflow` (Python, для LangChain deepagents) и QwenLM/qwen-code #4721 (TypeScript, для qwen-code). Известные регрессии Opus 4.8 (issues #64217, #64774, #64991) — это важный caveat, для стабильности в long-running workflow пока рекомендуют 4.7 или Sonnet 4.6.
- **Применение для AstroFinSentinelV5:** Dynamic Workflows — это ровно тот паттерн, которого не хватает нашему фиксированному пайплайну `router → 5 параллельных агентов → synthesis → risk`. Вместо hard-coded HYBRID_WEIGHTS можно представить **adversarial synthesis**: 16 sub-агентов генерируют разные стратегии (консервативный, агрессивный, contrarian, астро-усиленный, hedge-стратегия), затем 3 judge-агента оценивают их через разные lens (Sharpe, max drawdown, астро-консистентность) и только лучший результат идёт в `TradingSignal.from_agents()`. Опыт `langchain-dynamic-workflow` показывает, что это переносимо в Python-стек без миграции на Claude Code — нужно только поднять «meta-layer» скрипт-оркестратор рядом с текущим `synthesis_agent.py`. Кроме того, **CaveAgent-style persistent runtime** + Dynamic Workflows дают двухуровневую защиту: stateful Python-runtime хранит DataFrame с позициями между ходами (экономия ~28% токенов), а Dynamic Workflows фан-аутят по горизонтали — отличный стек для R-09 (AgentResponse) и ATOM-KARL-010 (ContinuousBacktest) при long-running research-loop. Важный риск: известные регрессии Opus 4.8 в tool-call парсинге, поэтому при production-внедрении стоит пока держать Opus 4.7 или Sonnet 4.6 как fallback.

- **Краткое описание:** Microsoft выкатил **полный enterprise-стек для агентов** на Build 2026: (1) **Windows Agent Framework** — open-source под MIT для .NET и Python (production-ready обвязка: Agent Harness, CodeAct, Handoff pattern, MCP tool discovery, A2A cross-runtime); (2) **Microsoft Execution Containers (MXC) v0.6.1** — policy-driven runtime containment на Rust, новая защитная граница для автономных агентов (где они могут исполнять код, модифицировать файлы, вызывать tools); (3) **Azure Agent Mesh** (GA Q4 2026) — managed-оркестрация агентов с Entra identity, autoscaling, hosted agents, session state, observability, versioned deployments и traffic splitting; (4) **Agent Governance Toolkit v4.0.0** — отдельный open-source пакет с policy engine, capability model, audit, MCP gateway, identity, trust scoring, TEE keystore, Entra-signed JWT verification, capability grant auth, NSAMCP compliance mapping, AGT Studio; (5) **GitHub Copilot for Azure** + **azd** CLI для deployment, Foundry Observability с OpenTelemetry в Application Insights. Параллельно JetBrains и OWASP выпустили свои фреймворки (Agentic AI Security Maturity Framework), Cloud Security Alliance — ORCHIDEAS + MAESTRO threat modeling.
- **Применение для AstroFinSentinelV5:** Этот стек закрывает сразу 3 TODO из нашего `AGENTS.md`: (a) **MXC** даёт готовую runtime-изоляцию для нашего sandbox (вместо ad-hoc Docker), (b) **Agent Governance Toolkit v4.0.0** формализует Conflict Resolution таблицу — capability grants могут заменить текущий «Astro vs Fundamental+Quant» rule с явной policy engine, (c) **Hosted Agents в Foundry** — направление миграции с SQLite + локального Python на managed-хостинг с готовой observability. Самое ценное — **AGT v4.0.0 включает LangGraph v1.0 governance adapter**, а наш `KARLSynthesisAgent` построен на LangGraph StateGraph, поэтому интеграция будет особенно прямой. **OWASP Agentic AI Security Maturity Framework** (4 уровня maturity) — это готовый чеклист, по которому можно аудировать наш `core/risk.py` и `core/volatility.py` уже сейчас. Известная .NET-регрессия (issue #6231) в hosted Foundry agent — сигнал, что migration лучше делать в два этапа: сначала policy-enforcement через AGT, потом уже managed hosting.

**3. arXiv/ACL ARR 2026 — R-HAN, Failing Tools benchmark и SkillsVote: фундаментальные исследования для production-grade агентов**

- **Источник:** OpenReview / arXiv (R-HAN: wmucOCOBPq, Failing Tools: j7YsSnA64D, SkillsVote: kj068rI9Uh, EAPRouter: c9mKOXi2zV, R_FOLD: Wlz2pfZwEu, MA-GRL: s41598-026-56204-2)
- **Краткое описание:** Несколько препринтов за последнюю неделю дают серьёзный conceptual backbone для production multi-agent систем: (a) **R-HAN** (Reliable Hierarchical Coordination) — union graph из MCTS + DPP выбранных топологий + Hierarchical Sparse Coordination (сжатые latent briefs между агентами) + Local Self-Refinement; SOTA на 8 бенчмарках, +2.98% в среднем, до +10.27% на MMLU-Pro; (b) **Failing Tools: Benchmarking LLM Agent Recovery Under Runtime Tool Failures** — канонический benchmark с 218 сценариями runtime-сбоев (denial, staleness, silent no-ops, cascading errors), результаты: ни одна frontier-модель не превышает 11.47% accuracy на recovery-evaluator, главный gap — отсутствие verification/recovery steps, а не неправильный tool selection; (c) **SkillsVote** — governance layer для agent skills: коллекция → рекомендация → атрибуция → evidence-gated evolution, улучшает Terminal-Bench 2.0 и SWE-Bench Pro; (d) **EAPRouter** (Grounding LLM Agent Intent in Multi-Module Environments) — executable action programs как IR для grounding намерений агента, +31.6% над function-calling baseline, safety validation блокирует все prompt-injection попытки; (e) **R_FOLD** — bi-level context management для long-horizon агентов (81.33% accuracy при 32K token budget); (f) **MA-GRL** — decentralized graph-attention MARL для urban traffic routing с implicit coordination reward (11.1% улучшение travel time, 87% zero-shot transfer между городами).
- **Применение для AstroFinSentinelV5:** **Failing Tools** — самый практически полезный benchmark: показывает, что наш SynthesisAgent и QuantAgent (полагающиеся на 5+ внешних tools) могут падать молча — добавить в `core/checkpoint.py` обязательный verification step (например, после `get_market_data()` вызвать `verify_freshness()`) — это напрямую ложится на ATOM-KARL-010 KPI Control Loop. **R-HAN** даёт формальный путь к **sparse coordination в SynthesisAgent**: вместо того чтобы все 5+ агентов всегда голосовали с полным весом, можно сжимать их сигнал в latent brief (по аналогии с `from_agents(astro_weights=...)`), адаптивно активировать только те, что дают дифференцирующий сигнал, и через contrastive evidence-rewriter доводить решения. **SkillsVote governance** идеально ложится на наш ATOM-009/010 audit trail — `DecisionRecord` уже хранит top_trajectories и final_action, нужно только добавить evidence-атрибуцию («какой skill-агент привёл к этому решению») и gates на admission. **EAPRouter** даёт идею **grounded action programs** для QuantAgent — вместо свободных tool-calls пишется typed Python-IR, который проходит AST-validation, что отлично дополнит `@require_ephemeris` декоратор и даст защиту от prompt-injection в финансовом контексте. **R_FOLD** — практический рецепт для long-horizon monitoring: при watch-list сессии (мульти-актив, недели) использовать bi-level context management (summarize+retain) вместо наивного sliding window.
