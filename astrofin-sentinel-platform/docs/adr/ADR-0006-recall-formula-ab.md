## Context

ADR-0005 introduced a relation-weight axis. With it, the recall-score
formula became:

```
recall_score = tier_weight Ã— decay Ã— confidence Ã— relation_weight
```

But there is no evidence yet that this exact functional form is optimal.
We need to test alternatives empirically before treating v1 as final.

We also want to be able to A/B-test candidate formulas **without
rewriting** `infer_edges.py` every time, and against a reproducible
ground-truth set rather than just the override contract.

## Decision

### 1. Pluggable formulas

`infer_edges.py` now exposes a `compute_recall_score(tier, decay, conf,
rel_weight, formula)` function and a `--score-formula` CLI flag
with four variants:

| Variant | Formula | Notes |
|---------|---------|-------|
| `v1_baseline` | `tier Ã— decay Ã— conf Ã— rel` | current default (ADR-0005) |
| `v2_no_tier` | `decay Ã— conf Ã— rel` | drops tier axis |
| `v3_smooth_tier` | `âˆštier Ã— decay Ã— conf Ã— rel` | softer tier penalty |
| `v4_relation_primary` | `rel Ã— decay Ã— conf Ã— 0.4 + 0.6 Ã— tier` | linear blend |

### 2. Ground truth

`graphify-out/ground_truth.jsonl` (24 edges) generated by
`graphify-out/build_ground_truth.py` following principle (c):

- 7 `override_applied=true` edges (label=1, ADR-0004 source-of-truth)
- 7 cross-file T1 edges with high `recall_score` (label=1)
- 10 T3 background edges (label=0)

This is a **starter** GT, not a hand-curated gold set. It is sufficient
to detect formula regressions but not to rank fine-grained weight
differences. Future work: grow to ~50 hand-verified edges.

### 3. A/B runner

`graphify-out/run_ab_calibration.py` evaluates each formula against
`inferred_clean.enriched.jsonl` (316 edges) and writes
`graphify-out/ab_calibration_v3.json`.

## Conclusion

**Keep `v1_baseline` as default.**

- `v2_no_tier` breaks the ADR-0004 override contract (only 2/7 in
  top-10, 3/7 in top-20). Rejected.
- `v3_smooth_tier` is mathematically equivalent to `v1` on the current
  sample (T1 edges have `confidence = 1.0`, `decay â‰ˆ 1.0`, so
  `âˆštier` collapses to the same value as `tier` for the discrete
  tier set). No benefit; no harm; left as future option.
- `v4_relation_primary` matches `v1` on recall quality but introduces
  a linear blend that hides the multiplicative semantics that downstream
  code relies on. Rejected.

## Open issues

1. **Override tie-break regression.** With 13 edges tied at
   `recall_score=1.0` (all `calls`), only 5 of 7 override edges land in
   the top-10 under stable sort. ADR-0004 implicitly assumes overrides
   always reach the top. This is **not** a formula problem â€” it is a
   **secondary-sort policy** problem. Tracked as candidate ADR-0007:
   "override-aware secondary sort".
2. **Monomorphic top-10.** All top-10 edges under v1 are `calls`.
   Improving relation diversity in top-K requires either (a) more
   diverse input relations at high tier, or (b) a relation-diversity
   re-ranker. Out of scope for this ADR.
3. **Top-20 relation mix** still skews `calls:13, contains:7`. With
   ground-truth growth, future A/B can detect whether `contains`
   should be demoted relative to `uses`/`imports_from`.
